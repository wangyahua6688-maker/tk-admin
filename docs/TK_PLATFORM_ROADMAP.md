# TK 平台开发路线图（修订版）

更新时间：2026-03-03

## 一、架构原则

1. 页面形态：做主流 Web/WebApp 自适应页面，是否使用 SPA 不是安全前提。
2. 安全体系：独立建设 API 安全、传输安全、鉴权与审计，不依赖前端形态。
3. 模块分层：按业务域拆分，不再使用“单一 business 模块”聚合所有职责。
4. 数据驱动：`tk-web`、`tk-api`、`tk-admin-web`、`tk-admin` 全部接入数据库。

## 二、业务模块拆分（最终目标）

### 1) 客户端用户管理域
- 客户端用户列表
- 发帖管理
- 热点评论

### 2) 业务配置域
- Banner 配置：广告 + 官方通知（同一张表，`type` 区分）
- 系统广播配置
- 开奖信息配置（彩种按钮区 + 期开奖内容 + 竞猜选项）
- 官方发帖管理（含评论查看）
- 外链配置

### 3) 客户端展示域
- 首页：banner轮播、系统广播、澳彩/港彩切换、条件直播、开奖展示
- 彩种列表：图卡式入口
- 彩种详情：年份切换、期数切换、大图、竞猜进度条

## 三、数据库设计（`tk_` 前缀）

已建立/接入的核心表：
- `tk_users`
- `tk_banner`
- `tk_broadcast`
- `tk_special_lottery`
- `tk_lottery_info`
- `tk_lottery_option`
- `tk_post_article`
- `tk_comment`
- `tk_external_link`

说明：
- `tk_banner` 使用 `type` 区分广告和官方；使用 `position` 区分首页、彩种详情、帖子详情等展示区。
- `tk_special_lottery` 维护首页按钮切换区及直播状态字段（是否开播、流地址）。

## 四、安全防护基线

### API 层
- JWT 鉴权
- 请求签名（可开关，HMAC + 时间戳 + nonce）
- IP 限流
- 请求体大小限制
- 请求 ID 全链路追踪
- 安全响应头（CSP/XFO/Referrer 等）

### 传输层
- 生产必须 HTTPS
- WebSocket Origin 白名单校验

### 后台层
- RBAC 权限码按业务域细分
- 操作审计与登录日志

## 五、开发优先级

### P0（已完成）
1. `tk-web`：白底现代自适应首页，双按钮开奖区，直播条件显示，图卡列表，彩种详情页骨架。
2. `tk-api`：MySQL 接入 + `tk_` 表迁移 + 种子数据 + 首页/彩种/论坛接口改为读库。
3. `tk-admin`：拆分为 `user-mgmt` 与 `biz-config` 路由域，读 `tk_` 表。
4. `tk-admin-web`：菜单与页面按两大域拆分，并接入新 API。

### P1（下一阶段）
1. 为 `tk-admin` 增加完整 CRUD（当前以列表查询为主）。
2. 接入论坛评论树、追评与官方发帖评论管理。
3. 完成彩种详情投票写接口与防刷票策略。
4. 打通后台配置改动 -> 客户端实时生效链路。

### P2（高并发与稳定性）
1. Redis 缓存首页聚合与热点数据。
2. Kafka 事件化（开奖、帖子、广播变更）。
3. WebSocket 广播改为 Redis Pub/Sub 多实例分发。

### P3（治理）
1. 设备指纹与风控。
2. 密钥轮换与配置中心。
3. 监控告警与压测基线。
